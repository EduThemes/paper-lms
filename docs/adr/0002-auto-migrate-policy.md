# 0002. `AUTO_MIGRATE=false` in prod / CI; SQL chain as source of truth

## Status

Accepted

## Context

GORM's `AutoMigrate` is a productivity multiplier when you're iterating
on a model: change the struct, restart the server, and the column
appears. But two things make it dangerous past prototype:

1. **AutoMigrate and a hand-written SQL migration chain produce
   different schemas.** GORM `uniqueIndex` tags emit `uni_*` constraint
   names; a SQL migration emits `idx_*` shapes. Acronym fields
   (`OIDC*`, `IDP*`) get autogenerated column names (`o_id_c_*`) that
   don't match the migration column names unless you add an explicit
   `gorm:"column:..."` tag. A dev DB that ran BOTH accumulates both
   shapes and masks the drift.
2. **Some constraints can't be expressed in GORM at all.** Partial
   UNIQUE indexes (`WHERE pseudonym_name IS NOT NULL`), append-only
   triggers on `audit_logs`, the pgvector extension, Postgres enum
   types backing our gamification scope/audience columns — these
   exist only in the SQL chain.

The Phase 7-A audit landed the soft-fail behavior (AutoMigrate logs
mismatches and continues rather than killing the server) but the
underlying policy is firm: the SQL chain in `internal/db/migrations/`
is the source of truth.

## Decision

**`AUTO_MIGRATE=false` in production and CI. `AUTO_MIGRATE=true` is a
dev convenience only.**

- Production deployments and the CI test environment set
  `AUTO_MIGRATE=false` and run versioned migrations via
  `make migrate-up` (which invokes `cmd/migrate`, a wrapper over
  `golang-migrate` with embedded SQL).
- The migration chain in `internal/db/migrations/NNNNNN_*.up.sql` is
  authoritative. Adding a model **requires** the matching migration in
  the same PR.
- `TestSchemaParity_Wave1` and `TestSchemaParity_Wave3` are CI
  hard-fails on stale columns, missing columns, and missing indexes.
  Local tools `make schema-diff` (what AutoMigrate builds that the
  SQL chain doesn't) and `make stale-cols` (the reverse) catch drift
  before push.
- The two-migration rename pattern (Wave A copy → Wave B drop) is the
  only supported way to rename or remove a column. See
  `CONTRIBUTING.md` for the procedure.

## Consequences

- **Dev DBs older than 2026-05-16 must be blown away and re-migrated
  from scratch**; they carry the union of AutoMigrate's column names
  and the SQL chain's, which hides bugs CI then catches.
- **GORM tags carry load-bearing metadata**: explicit `gorm:"column:..."`
  on acronym fields, `gorm:"type:text"` on enum-backed columns so
  AutoMigrate doesn't try to recreate the Postgres enum, `BeforeCreate`
  hooks that mirror SQL `DEFAULT` values for `NOT NULL` columns. These
  tags are not stylistic.
- **Nullable text columns with partial UNIQUE indexes MUST be `*string`
  in Go, not `string`.** GORM serializes `""` as the empty string (not
  `NULL`), which IS indexed by a `WHERE col IS NOT NULL` partial index
  and collides on the second unassigned row.
- The dev workflow keeps iteration-friendly: change a struct, restart,
  see it work. When the migration story is real, run `make schema-diff`,
  author the SQL file via `make migrate-create`, and confirm parity.
